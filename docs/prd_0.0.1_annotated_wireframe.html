MVP 评审结论应确认：
  1. Retag 是默认路径，允许 Codex running 但提示刷新 / 重启后 UI 更稳定。
  2. Clone 和 batch clone 必须在 Codex stopped 时才能 Apply，因为会写全局状态索引。
  3. 所有写操作前创建 snapshot；批量迁移每批只创建一个 batch snapshot。
  4. 确认框必须把 selected、archived、subagent、mode、writes、rollback 条件一次讲清。
  5. 验收用例以 019e6717-XXXX 会话作为单条 retag / rollback 主路径。
信息架构
  1. Header 展示当前环境状态。
  2. 左侧 provider，右侧 session。
  3. 下方 detail 承接当前光标项。
  4. Actions 固定提示快捷键。
默认过滤
  1. archived = 0。
  2. source = vscode。
  3. thread_source = user。
  4. subagent 默认不显示。
安全分层
  1. Retag：低风险，默认推荐。
  2. Clone：高风险，需要二次确认。
  3. Batch：必须展示汇总与失败策略。
写入边界
  1. 不迁移 auth.json。
  2. 不修改 API key。
  3. 不读取或打印 token。
  4. 不自动关闭 Codex Desktop。
校验规则
  1. DB provider 已变更。
  2. rollout 首行 provider 已变更。
  3. rollout 文件存在。
  4. pragma integrity_check = ok。
占位约定
  • 数量使用 XX / XXXX。
  • 日期使用 YYYY/MM/DD。
  • 保留验收 thread 前缀，便于评审定位。
